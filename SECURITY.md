# 安全策略

Nexora 目前处于 early alpha，尚未承诺稳定版本支持周期。安全修复会优先合入当前开发分支，
必要时回移到最新的 `0.x` 版本；更早的预发布版本通常不会单独维护。

| 版本 | 支持状态 |
| --- | --- |
| 当前主分支 | 尽力支持 |
| 最新 `0.x` 版本 | 尽力支持 |
| 更早版本 | 不支持 |

## 私下报告漏洞

请不要在公开 Issue、Discussion、Pull Request 或日志中披露漏洞细节、利用代码、凭据或用户数据。

1. 首选仓库 **Security** 页面中的 **Report a vulnerability**，通过 GitHub Private
   Vulnerability Reporting 提交；
2. 如果仓库尚未显示该入口，请创建一个不含任何技术细节的 Issue，标题使用
   `[security contact request]`，仅请求维护者提供私密沟通渠道；
3. 收到私密渠道后，再提供受影响版本、复现条件、影响范围、最小复现和可能的缓解方案。

维护者会确认收到报告、评估影响并协调修复与披露时间。early alpha 阶段暂不承诺固定响应时限，
但会尽快处理可能导致代码执行、凭据泄露、认证绕过、更新链路劫持或数据破坏的问题。

## 报告范围

适合私下报告的内容包括：

- Nexora 框架或 CLI 中可由不可信输入触发的安全问题；
- 桌面认证、Deep Link、自动更新或密钥存储边界中的缺陷；
- 示例服务端中可能误导使用者形成不安全部署的框架级默认行为；
- 仓库、发布产物或构建流程中意外暴露的凭据与敏感信息。

一般缺陷、功能建议和仅影响本地开发体验的问题可以使用公开 Issue。第三方依赖本身的漏洞请优先
报告给对应上游；如果 Nexora 的使用方式放大了风险，也请同时私下告知本项目。
