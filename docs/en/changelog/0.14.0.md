# Nexora 0.14.0

Release date: 2026-07-21

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.14.0`. This release adds unified Account avatar upload, user avatar update,
and ZITADEL metadata synchronization.

## Added

- Account contracts now include the `avatar_url` user-provisioning field, `AvatarUploadResponse`,
  and `UpdateUserAvatarRequest`; HTTP fields continue to use `snake_case`. Owner:
  [@zhutao525](https://github.com/zhutao525)
- Account server APIs now include the `AvatarStorage` abstraction, `LocalAvatarStorage`, avatar
  upload, and user avatar update. Uploads are limited to 2 MiB, support PNG, JPEG, WebP, and GIF,
  and returned URLs must be HTTPS or loopback HTTP. Owner: [@zhutao525](https://github.com/zhutao525)
- Account HTTP now exposes `POST /avatars` and `PATCH /users/{user_id}/avatar`, with matching OpenAPI
  documentation. Owner: [@zhutao525](https://github.com/zhutao525)
- The ZITADEL directory integration now writes, reads, and updates avatar metadata using the stable
  key `urn:nexora:account:avatar_url`. Owner: [@zhutao525](https://github.com/zhutao525)

## Changed

- Provisioned users can now receive an avatar URL that is stored in both ZITADEL metadata and the
  local Account user. When a ZITADEL profile has no avatar, Account falls back to the Nexora metadata
  value. Owner: [@zhutao525](https://github.com/zhutao525)
- User avatar updates synchronize the identity directory before updating the local database. If the
  local update fails, Account best-effort restores the previous identity-directory avatar and logs any
  compensation failure. Owner: [@zhutao525](https://github.com/zhutao525)
- The default desktop user-provisioning form can collect an avatar URL, and the Account desktop client
  exposes avatar upload and avatar update methods. Owner: [@zhutao525](https://github.com/zhutao525)
- Updated the root workspace version, internal workspace dependency constraints, CLI installation
  examples, and OpenAPI metadata to `0.14.0`. Owner: [@zhutao525](https://github.com/zhutao525)

## Fixed

- Corrected the newly added avatar client rustdoc encoding and completed the public API error
  documentation for avatar updates. Owner: [@zhutao525](https://github.com/zhutao525)

## Compatibility and Breaking Changes

- `ProvisionUserRequest` adds the optional `avatar_url` field. Existing JSON requests do not need to
  change.
- Added the `users:avatar.write` permission. The built-in admin role receives it through migration;
  custom roles need an explicit grant before they can upload, update, or clear user avatars.
- Added migration `0009_account_add_user_avatar_permission`; upgraded deployments must run Nexora
  migrations. Rolling back to `v0.13.0` requires rolling back that permission and related role grants.
- Hosts that use avatar upload must configure `AccountDependencies::avatar_storage`; otherwise the
  upload endpoint returns service unavailable.
- ZITADEL avatar synchronization depends on UserService metadata support. Unsupported providers return
  a stable directory error instead of being silently ignored.

## Upgrade

1. Update the Nexora tag from `v0.13.0` to `v0.14.0` and refresh the lockfile.
2. Run database migrations so `users:avatar.write` and the default admin grant are applied.
3. Configure `AccountDependencies::avatar_storage` for hosts that need avatar uploads, for example
   with `LocalAvatarStorage::new(directory, public_base_url)`.
4. Grant `users:avatar.write` to custom roles that should manage user avatars.
5. Clients can call `POST /avatars` to obtain an `avatar_url`, then call
   `PATCH /users/{user_id}/avatar` to set or clear it. User provisioning can also pass `avatar_url`
   directly.
