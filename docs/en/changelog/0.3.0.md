# Nexora 0.3.0

Released on 2026-07-18.

This release completes the Account composition boundary for host application routers and turns the
default desktop user, role, and permission pages from read-only views into a practical management
surface. Applications can obtain an Account handle from an initialized `Server`, place it in a
custom Axum State, and reuse Nexora authentication and permission extractors directly. User
provisioning can also establish initial role grants in the same transaction.

## Added

- Added `Server::account()`, which returns a cloneable `Account` handle after
  `Server::initialize` succeeds. The handle reuses the application's one `PgPool` and never creates
  a second pool. It returns `None` before initialization. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- `AuthenticatedUser` and `Authorized<P>` now work with any custom Axum State satisfying
  `Account: axum::extract::FromRef<AppState>`. Hosts no longer need to copy token verification,
  local status checks, or merged-permission logic. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added pool-first `create_user_with_roles` and `Account::provision_user_with_roles`. They provision
  an already-confirmed `ExternalIdentity`, retain the built-in `member` role, and atomically record
  initial business roles with a local `granted_by` user ID. An invalid role or grantor rolls back
  the whole transaction. — Owner: [@Jixiangup](https://github.com/Jixiangup)

## Changed

- `ProvisionUserRequest` now has `role_ids`. The default `POST /users` endpoint can establish
  initial role grants while provisioning a user. Starting in 0.3.0, an empty set also grants the
  permissionless built-in `member` role, so effective permissions remain empty while the
  `user_roles` snapshot gains a relation compared with 0.2.0. A non-empty set requires both
  `users:provision` and `users:roles.write`. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- The default `/users` page now refreshes the user list, provisions confirmed OIDC identities,
  selects initial roles, activates or suspends regular users, and completely replaces a regular
  user's direct role set. Super-administrator and last-active-administrator protections remain
  enforced by the Account server. — Owner: [@Jixiangup](https://github.com/Jixiangup)
- The default `/roles` page now creates custom roles with initial permissions, updates role names
  and descriptions, replaces complete permission sets, and deletes custom roles. Creating a role
  and its initial permissions continues to use `roles:write`; built-in roles remain immutable. —
  Owner: [@Jixiangup](https://github.com/Jixiangup)
- Default management actions are gated by `users:provision`, `users:status.write`,
  `users:roles.write`, `roles:read`, `roles:write`, and `permissions:read`, without hard-coded role
  names in the desktop client. — Owner: [@Jixiangup](https://github.com/Jixiangup)

## Compatibility and breaking changes

### Custom Axum State

Existing 0.2.0 applications that only merge `server.routers()` can remain unchanged. An application
only needs the following composition when its own routers must reuse Account authentication or
permission extractors:

```rust
use axum::extract::FromRef;
use nexora::server::Account;
use sqlx::PgPool;

#[derive(Clone)]
struct AppState {
    pool: PgPool,
    account: Account,
}

impl FromRef<AppState> for Account {
    fn from_ref(state: &AppState) -> Self {
        state.account.clone()
    }
}

server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
let account = server.account().expect("Server has been initialized");
let state = AppState {
    pool: pool.clone(),
    account,
};

let app = axum::Router::new()
    .merge(server.routers())
    .merge(application_routes())
    .with_state(state);
```

Business handlers can then extract `AuthenticatedUser` directly or implement `RequiredPermission`
and use `Authorized<P>`. `authorization.profile().user.id` is the current Nexora-local user ID and
can be used for business audit columns. The extractors do not expose the bearer token.

### `ProvisionUserRequest` source and JSON compatibility

- Rust callers constructing `ProvisionUserRequest` with a struct literal must add
  `role_ids: Vec::new()` or actual role IDs. This is a source migration even though the JSON
  contract remains backward compatible.
- `role_ids` defaults to an empty collection during deserialization, so JSON produced by a 0.2.0
  client without the field is still accepted by a 0.3.0 server. The 0.3.0 server grants the
  permissionless built-in `member` role for this request: effective permissions remain empty, but
  the role-relation snapshot is no longer identical to 0.2.0.
- A 0.3.0 client omits an empty `role_ids` field, so the original provisioning request remains
  acceptable to a 0.2.0 server. Non-empty `role_ids` require both a 0.3.0 client and server.
- Non-empty `role_ids` require both `users:provision` and `users:roles.write`. Supplying non-empty
  `permission_ids` while creating a role continues to require only the existing `roles:write`.

### Database migration

Nexora 0.3.0 adds, removes, or rewrites no DDL and introduces no migration version. Existing 0.2.0
databases and `_sqlx_migrations` history require no manual data migration. Applications continue to
use the single composed Migrator introduced in 0.2.0.

## Upgrade from 0.2.0

1. Pin Nexora to `v0.3.0` and update the lockfile:

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.3.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

2. Add `role_ids` to every `ProvisionUserRequest` struct literal. Use `Vec::new()` when initial
   business roles are not needed.
3. When application routers need Account extractors, call `server.account()` after successful
   `server.initialize(...)`, store the handle in the final State, and implement
   `FromRef<AppState> for Account`. Applications using only Nexora's default routers can skip this.
4. Before using initial role assignment, ensure operators have both `users:provision` and
   `users:roles.write`. Role creation and its initial permissions continue to use `roles:write`.
5. Verify legacy empty-role provisioning, provisioning with initial roles, user status changes,
   user-role replacement, role management, and permission replacement.
6. Reinstall the 0.3.0 CLI:

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.3.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## Rollback notes

- This release has no schema changes, so the database needs no down migration. Still verify the
  application-code and lockfile rollback in pre-production.
- Before returning to 0.2.0, remove calls to `Server::account`, the generic Account extractors, and
  `create_user_with_roles`, and restore 0.2.0 `ProvisionUserRequest` struct literals.
- A 0.2.0 server does not understand non-empty `role_ids`. During rollback, disable initial-role
  provisioning or provision with the old request first and then use the existing user-role
  replacement endpoint.
