# Nexora 0.11.0

Release date: 2026-07-19

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.11.0` without a beta suffix. Because this release adds public server APIs
and changes the public `VerifiedIdentity` struct shape, it bumps the minor version in the `0.x` line.

## Added

- Added `nexora::server::ZitadelProvisioningClient` for dynamic ZITADEL Organization lookup/creation,
  portal Project Grant creation/update, human-user creation/deactivation/deletion in a target
  Organization, and portal project authorization create/update/delete operations. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added `Server::zitadel_provisioning_client()` and
  `nexora::server::provisioning_client(&settings)`, reusing the existing Account server issuer, PAT,
  TLS, loopback-development, and sensitive-log-redaction rules. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added independent OIDC resource-server authentication APIs:
  `OidcAccessTokenVerifier::discover_many`, `OidcResourceServer`, `BearerAccessToken`, and
  `VerifiedBearerIdentity`, so customer portals and public APIs can verify independent audiences
  without binding to Nexora Account's local user table or RBAC. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added `organization: Option<VerifiedOrganizationContext>` to `VerifiedIdentity`, preserving the
  ZITADEL Organization id, name, and primary domain from verified access tokens so business modules
  can map customer or portal tenants. — Owner: [@Jixiangup](https://github.com/Jixiangup)

## Changed

- `ZitadelUserDirectory` now reuses the shared ZITADEL gRPC channel/PAT boundary, so Account user
  management and portal provisioning do not duplicate TLS, loopback, or sensitive-log rules. —
  Owner: [@Jixiangup](https://github.com/Jixiangup)
- The local minimal ZITADEL proto contracts now include the Organization, Project Grant,
  Authorization, and User deactivate RPCs/messages required by the framework provisioning client. —
  Owner: [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- Authentication errors, configuration errors, and gRPC client Debug output continue to avoid echoing
  access tokens, PATs, or provider internals. New tests cover portal verifier error mapping and PAT
  Debug redaction. — Owner: [@Jixiangup](https://github.com/Jixiangup)

## Compatibility and Breaking Changes

- Breaking change: the public `VerifiedIdentity` struct has a new `organization` field. Downstream
  tests or custom `AccessTokenVerifier` implementations that construct `VerifiedIdentity` with a
  struct literal must add:

```rust
organization: None
```

- Feature compatibility: downstream projects still only use Nexora's `server` / `desktop` features;
  they do not need to, and should not, enable the internal `zitadel` feature directly.
- Default Account `/users`, `/roles`, `AuthenticatedUser`, and `Authorized<P>` behavior remains
  compatible. Customer portal users are not mixed into the default internal Account management pages.
- No database migration, OpenAPI route, default Account configuration, or desktop UI behavior change.
  Rolling back to `v0.10.0` requires removing calls to the new provisioning/resource-server APIs and
  reverting `VerifiedIdentity` struct-literal fields.

## Upgrade

1. Update the Nexora tag from `v0.10.0` to `v0.11.0` and refresh the lockfile.
2. Search for `VerifiedIdentity {` and add `organization: None` or a verified
   `VerifiedOrganizationContext` to struct literals.
3. To provision customer portals, call `server.zitadel_provisioning_client()` after
   `Server::initialize`, or create one with `nexora::server::provisioning_client(&settings)?`.
4. A customer-create flow should find or create a ZITADEL Organization, include the platform service
   account or chosen administrator in `administrator_user_ids`, then call `ensure_project_roles`,
   `ensure_project_grant`, `create_human_user`, and `ensure_authorization`. Use `delete_user` for
   compensation if business-database writes fail after creating a provider user.
5. Portal and public API backends should create an `OidcResourceServer` with
   `OidcAccessTokenVerifier::discover_many(issuer, audiences)`, store it in Axum state, and use
   `VerifiedBearerIdentity` to read `identity.organization.id` for tenant mapping instead of using
   Account's local-user/RBAC extractors.
6. No generated-project Skill sync or database migration is required. This release does not change
   the default generated project structure.
