# Nexora 0.12.0

Release date: 2026-07-20

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.12.0`. This release adds initial-password support when Nexora Account
creates human users. As requested, the implementation keeps the public API direct and intentionally
breaks the previous password-less request shape, so callers must update their Rust struct literals or
HTTP request bodies when upgrading from `0.11.3`.

## Added

- Account can now create human users with `initial_password` and `require_password_change`.
  The ZITADEL UserService v2 create path writes the initial password to the external identity
  directory while Nexora continues to store only the local user binding and role grants. Passwords are
  not stored in local plaintext fields, logs, or error details. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- The HTTP/client `ProvisionUserRequest` contract now has a required `initial_password` field and an
  optional `require_password_change` field, so iMES can create employee accounts with the phone number
  as the username and `imes{phone}.` as the default initial password. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Changed

- `CreateHumanIdentity` now directly includes `initial_password` and `require_password_change`.
  Host applications calling `nexora::server::Account::create_managed_user_with_roles` must provide an
  initial password. — Owner: [@Jixiangup](https://github.com/Jixiangup)
- The default Account user-creation UI now generates an initial password for the existing manual flow
  that has no password input, and requires the user to change it after first login. Business modules
  should still pass their own initial-password rule through the backend facade. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Updated the root workspace version, internal workspace dependency constraints, CLI installation
  examples, and OpenAPI metadata to `0.12.0`. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Issues and PRs

- Issue: none. This release comes from the iMES employee-creation flow needing Nexora Account to write
  the initial password into the authentication system.
- PR: none; committed and released directly from the default branch as requested.

## Compatibility and Breaking Changes

- Public API breaking change: `CreateHumanIdentity` and `ProvisionUserRequest` now require
  `initial_password`. Old password-less creation requests no longer compile or pass server
  validation.
- Old Rust shape:

  ```rust
  CreateHumanIdentity {
      username,
      given_name,
      family_name,
      email,
      display_name,
  }
  ```

- New Rust shape:

  ```rust
  CreateHumanIdentity {
      username,
      given_name,
      family_name,
      email,
      display_name,
      initial_password,
      require_password_change: false,
  }
  ```

- Old HTTP request bodies did not include `initial_password`; new request bodies must include:

  ```json
  {
    "initial_password": "imes13800000000.",
    "require_password_change": false
  }
  ```

- No database schema, migration SQL, server configuration, or scaffold-rule changes are included. To
  roll back to `v0.11.3`, also remove the initial-password fields from downstream callers.

## Upgrade

1. Update the Nexora tag from `v0.11.3` to `v0.12.0` and refresh the lockfile.
2. Add `initial_password` to every business module that builds `CreateHumanIdentity` or
   `ProvisionUserRequest`.
3. iMES should use the phone number as `username` and pass the desktop-submitted or default
   `imes{phone}.` value as `initial_password`; set `require_password_change` according to the business
   policy.
4. No manual database migration, generated-project Skill sync, macro-rule update, or
   generated-project constraint update is required.
