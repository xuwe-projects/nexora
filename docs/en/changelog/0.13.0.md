# Nexora 0.13.0

Release date: 2026-07-20

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.13.0`. This release adds the permission visibility and implied-permission
capabilities requested by iMES.

## Added

- Feature metadata now supports client-side visible permissions, and the derive macro accepts
  `visible_permissions(any = ["employees:read"])`. Manually implemented Feature metadata can use
  `FeatureVisiblePermissions`, `with_visible_permissions`, or `with_visible_permissions_any`. This is
  navigation and route visibility metadata only; it does not replace server-side `Authorized<P>`
  checks. Owner: [@Jixiangup](https://github.com/Jixiangup)
- The desktop sidebar hides Features that are not visible to the current login profile, then hides
  NavigationGroups that no longer have any visible children. Anonymous users do not see protected
  Features, and super administrators always see them. Owner: [@Jixiangup](https://github.com/Jixiangup)
- Account permission registration now includes `PermissionCatalogDefinition` and
  `PermissionDefinition::with_implies` for declaring implied permissions. Existing
  `PermissionDefinition` and `create_permissions`/`register_permissions` callers remain compatible.
  Owner: [@Jixiangup](https://github.com/Jixiangup)

## Changed

- When creating a role or replacing role permissions, Account recursively expands registered
  permission implications, deduplicates and stably sorts the result, then stores the final set in
  `account.role_permissions`. For example, granting `employees:write` that implies `employees:read`
  stores both permissions. `/me` reads the stored final set and does not infer permissions at runtime.
  Owner: [@Jixiangup](https://github.com/Jixiangup)
- Added migration `0008_account_permission_implications` to store implication edges and seed built-in
  Account write-to-read implications. The migration backfills existing role grants so upgraded
  databases immediately follow the final-set storage semantics. Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- The default Account users and roles Features now declare `users:read` and `roles:read` visible
  permissions. Owner: [@Jixiangup](https://github.com/Jixiangup)
- Updated the root workspace version, internal workspace dependency constraints, CLI installation
  examples, and OpenAPI metadata to `0.13.0`. Owner: [@Jixiangup](https://github.com/Jixiangup)

## Issues and PRs

- Issue: none. This release comes from the iMES integration need for permission-aware desktop
  navigation and write permissions that imply read permissions.
- PR: none; committed and released directly from the current branch as requested.

## Compatibility and Breaking Changes

- Existing `PermissionDefinition` callers do not need to change. Permissions without `implies`
  declarations behave as before.
- The database gains a new `account.permission_implications` table. Upgrades must run Nexora
  migrations. Rolling back to `v0.12.0` also requires rolling back that table and the related role
  permission backfill.
- `visible_permissions` is only a client-side navigation and route visibility hint. Security
  boundaries must still use server-side `Authorized<P>` or equivalent authorization.

## Upgrade

1. Update the Nexora tag from `v0.12.0` to `v0.13.0` and refresh the lockfile.
2. iMES Features can declare `#[nexora(visible_permissions(any = ["employees:read"]))]` where needed.
3. iMES permission catalogs can register write-to-read implications with
   `PermissionDefinition { key: "employees:write", ... }.with_implies(["employees:read"])`.
4. Run database migrations so `account.permission_implications` and the role-permission backfill are
   applied.
