# Nexora 0.16.0

Release date: 2026-07-24

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.16.0`. This release adds role owner scopes so the iMES customer portal can
reuse Nexora Account/RBAC without copying role and permission tables into the business database.

## Added

- Account roles now have an `owner` field through the append-only `0010_account_role_owner_scope` migration, backfilled to the default `IMES` scope for backend/system roles. Owner: [@zhutao525](https://github.com/zhutao525)
- Added the global portal administrator system role `portal_admin`, with `owner = "IMES"` and `is_system = true`; hosts synchronize its permissions at startup, and normal role edit/delete endpoints cannot mutate it. Owner: [@zhutao525](https://github.com/zhutao525)
- Added owner-scoped trusted host APIs: `create_role_for_owner`, `create_generated_role_for_owner`, `replace_role_permissions_for_owner`, `replace_user_roles_for_owner`, `ensure_system_role_with_permissions`, and `grant_user_role`, re-exported from `nexora::server` and `nexora::account`. Owner: [@zhutao525](https://github.com/zhutao525)
- HTTP/OpenAPI contracts now include `owner` and default-`IMES` owner query/write semantics, while requests that omit `owner` keep the existing backend-system behavior. Owner: [@zhutao525](https://github.com/zhutao525)

## Changed

- The default `roles`, `role`, `create_role`, `replace_role_permissions`, and `replace_user_roles` semantics still target `owner = "IMES"` to preserve existing backend user-role management behavior. Owner: [@zhutao525](https://github.com/zhutao525)
- `replace_user_roles_for_owner` replaces only the user's roles in the requested owner and leaves `IMES` and other customer owners untouched; `grant_user_role` idempotently appends one role without clearing existing roles. Owner: [@zhutao525](https://github.com/zhutao525)
- Role keys remain globally unique. `create_generated_role_for_owner` uses the database sequence to generate globally unique `role_<id>` keys for portal roles that do not need human-entered codes. Owner: [@zhutao525](https://github.com/zhutao525)
- Updated the root workspace version, internal workspace dependency constraints, CLI installation examples, and OpenAPI metadata to `0.16.0`. Owner: [@zhutao525](https://github.com/zhutao525)

## Issues and PRs

- Issue: none. This release comes from the iMES customer-portal Account/RBAC integration need.
- PR: none; committed and released directly from the current branch as requested.

## Compatibility and Breaking Changes

- No breaking changes. Existing role creation, role-permission replacement, and user-role replacement requests that omit `owner` default to the `IMES` backend scope.
- `Role` / `RoleResponse` responses now include the additive `owner` field.
- The database migration adds `account.roles.owner TEXT NOT NULL DEFAULT 'IMES'` and keeps the global unique `roles.key` constraint.

## Upgrade

1. Update the Nexora tag to `v0.16.0` and refresh the lockfile.
2. Run the merged Nexora SQLx migrations and ensure `0010_account_role_owner_scope` has been applied.
3. iMES customer-portal roles should use the customer ID string as owner; backend system roles and `portal_admin` keep using `IMES`.
4. At host startup, register portal permissions first, then call `ensure_system_role_with_permissions(PORTAL_ADMIN_ROLE_KEY, ...)` to synchronize the global portal administrator permissions.
5. Use `grant_user_role` to bind the default portal administrator role, and use `replace_user_roles_for_owner` when replacing customer-scoped role assignments.
