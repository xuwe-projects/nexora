# Nexora 0.4.0

Released on 2026-07-18.

This release rebuilds the default user-management experience around a card-styled `DataTable` with
virtualized infinite scrolling, avatars, login usernames, and semantic status indicators. User
creation now uses a height-constrained standard `Dialog` with a fixed title, scrollable content,
and fixed footer. Account contracts and storage also gain an optional login username while the
stable `identity_id` remains the authentication binding key.

## Added

- Added optional `username` fields to `ExternalIdentity`, `ProvisionUserRequest`, `User`, and
  `UserResponse`. OIDC login synchronizes `preferred_username`, and administrators can bind a
  username during user creation. Older JSON requests may continue to omit it. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added SQLx migration `0006_account_add_username`, introducing a nullable, 200-character
  `account.users.username` column with complete database comments. Existing users remain unbound
  instead of receiving an unreliable inferred value. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Embedded a `rotate-ccw` icon in the desktop runtime asset source for the user-list refresh action.
  — Owner: [@Jixiangup](https://github.com/Jixiangup)
- Added complete Chinese and English `nexora::server` references. The HTTP reference documents route
  ownership, permissions, path/query/form/JSON inputs, response models, stable errors, and examples;
  the Rust reference documents the `Server` lifecycle, Account facade, pool-first functions,
  extractors, and Setup extension points. OpenAPI 3.1 now includes the default Setup SSR forms and
  Account contract. — Owner: [@Jixiangup](https://github.com/Jixiangup)

## Changed

- The default `/users` table now uses `gpui-component::DataTable`, stable user row IDs, virtualized
  rendering, and `TableDelegate::load_more` to fetch the next page near the bottom. Manual
  pagination has been removed. — Owner: [@Jixiangup](https://github.com/Jixiangup)
- User cells now show avatars, display names, local IDs, and a super-administrator marker. A
  separate login-username column replaces the previously exposed `identity_id` column. Status uses
  success/warning `Tag` components instead of plain text. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- The DataTable is hosted in the shared `Card`; `group_box.background` and `table.background` keep
  the card surface visually distinct from the desktop page, with theme border, radius, and shadow.
  — Owner: [@Jixiangup](https://github.com/Jixiangup)
- The primary action is now “Create user.” The form uses the standard `gpui-component` `Dialog`
  with a fixed title and footer; only content scrolls vertically when the dialog reaches its 640px
  maximum height. — Owner: [@Jixiangup](https://github.com/Jixiangup)
- The `publish-nexora-release` Skill and scaffold template now enforce SemVer ordering and choose
  the highest impact among breaking changes, compatible features, and fixes. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- Refresh atomically replaces rows from page one, while infinite loading only appends previously
  unseen user IDs. Status and role mutations explicitly refresh the DataTable state. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Compatibility and breaking changes

- HTTP JSON remains backward compatible: request and response usernames are optional, and
  `identity_id` authentication and uniqueness semantics are unchanged.
- Rust struct literals are source-incompatible. Callers constructing `ExternalIdentity`,
  `ProvisionUserRequest`, `User`, or `UserResponse` must add `username: None` or a trusted value.
  Nexora remains in initial `0.y.z` development, so strict SemVer advances MINOR to 0.4.0.
- The schema change only adds a nullable column and constraint. Apply migration 6 with the one
  composed SQLx Migrator before deploying 0.4.0 application code.

## Upgrade from 0.3.1

1. Pin Nexora to `v0.4.0` and update the lockfile:

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.4.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

2. Compose `nexora::server::migrations()` with application migrations and apply migration 6 with
   the existing single Migrator before deploying the new code.
3. Add `username` to Rust struct literals. Use `None` when no trusted username source exists; do
   not infer it from `identity_id`.
4. Custom Account clients may send optional `username` in `POST /users` and read it from user
   responses. Existing JSON clients do not need an immediate change.
5. Verify creation, avatar fallback, status tags, role management, refresh, and bottom-triggered
   continuous loading.
6. Reinstall the 0.4.0 CLI:

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.4.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## Rollback notes

- Keep the nullable `username` column when rolling application code back to 0.3.1. Its explicit
  queries ignore the extra column, and migration history remains intact.
- If a down migration is unavoidable, back up bound usernames and remove 0.4.0 field access first;
  the down migration permanently deletes username data.
- Remove the 0.4.0 field from affected Rust struct literals before compiling against 0.3.1.
