# Nexora 0.15.1

Release date: 2026-07-21

This release remains in the `0.x` beta line and does not move Nexora to `1.x`. The version uses the
standard SemVer value `0.15.1`. This release adds the ZITADEL human contact-phone behavior needed by
iMES employee creation and marks email and phone as verified during creation. `v0.15.1` is the
corrected release that includes the `v0.14.0` Account avatar release line; downstream projects should
not use the already pushed `v0.15.0`.

## Added

- `CreateHumanIdentity` now has the compatible construction path `with_contact_phone(phone)`, so callers can keep the existing struct fields while also writing the login phone number to ZITADEL human phone/mobile contact information. Owner: [@zhutao525](https://github.com/zhutao525)
- `IdentityDirectory` now includes the compatible default method `create_human_identity_with_contact`. Existing directory adapters ignore the extra phone, while the ZITADEL adapter writes non-empty phone numbers. Owner: [@zhutao525](https://github.com/zhutao525)
- `ZitadelUserDirectory::create_human_user_with_contact` and `ZitadelProvisioningClient::create_human_user_with_contact` support verified contact-phone creation in both default Account and customer portal provisioning flows. Owner: [@zhutao525](https://github.com/zhutao525)

## Changed

- ZITADEL UserService v2 `CreateUser` requests now set `SetHumanEmail.is_verified = true` instead of sending the default email verification code. When a contact phone is provided, requests set `SetHumanPhone.is_verified = true` and do not send an SMS verification code. Owner: [@zhutao525](https://github.com/zhutao525)
- The repository's minimal ZITADEL v2 proto contract now includes `SetHumanPhone`, `HumanPhone`, and `CreateUserRequest.Human.phone`. Owner: [@zhutao525](https://github.com/zhutao525)
- Updated the root workspace version, internal workspace dependency constraints, CLI installation examples, and OpenAPI metadata to `0.15.1`. Owner: [@zhutao525](https://github.com/zhutao525)

## Issues and PRs

- Issue: none. This release comes from the iMES integration need to store ZITADEL contact phones and skip email/SMS re-verification when creating employees.
- PR: none; committed and released directly from the current branch as requested.

## Compatibility and Breaking Changes

- No breaking changes. Existing `CreateHumanIdentity { ... }` struct literals and `Account::create_managed_user_with_roles(request, ...)` calls remain valid.
- If no contact phone is provided, or if it is blank, the ZITADEL create request omits the phone field and keeps the previous behavior.
- This release adds no database migration. The local Nexora Account database still stores only the identity profile snapshot and does not add a generic phone column.

## Upgrade

1. Update the Nexora tag to `v0.15.1` and refresh the lockfile; do not use the superseded `v0.15.0`.
2. iMES should keep using the employee phone number as `CreateHumanIdentity.username`, then call `.with_contact_phone(phone)` before passing the request to `Account::create_managed_user_with_roles`.
3. Keep the employee email in `CreateHumanIdentity.email`; the default ZITADEL implementation creates both email and contact phone as verified, so employees are not asked for another verification code.

Example:

```rust
let request = CreateHumanIdentity {
    username: phone.clone(),
    given_name,
    family_name,
    email,
    display_name,
    initial_password,
    require_password_change: false,
    avatar_url: None,
}
.with_contact_phone(phone);

account
    .create_managed_user_with_roles(request, role_ids, granted_by)
    .await?;
```
