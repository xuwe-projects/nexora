# Nexora 0.2.0

Released on 2026-07-17.

This release gives host applications direct Account-management and database-migration composition
boundaries. A host can use its one `PgPool` to create users, permissions, and roles and to maintain
their relationships. Nexora exports its migration list, while `Server::initialize` no longer
modifies the database implicitly. Removing the old migration entry points and changing the
initialization precondition make this a breaking release.

## Added

- Added five pool-first Account APIs under `nexora::server`: `create_permissions`, `create_role`,
  `replace_role_permissions`, `create_user`, and `replace_user_roles`. They reuse Account
  validation, stores, and transactions without creating another pool. Callers must be trusted host
  code and must authenticate and authorize the current operator themselves. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- `create_permissions` idempotently creates or updates the permission catalog; `create_role`
  writes a custom role and its complete initial permission set in one transaction; both
  `replace_*` APIs atomically replace complete relation sets; and `create_user` provisions only an
  externally verified identity without introducing local passwords. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)
- Added `nexora::server::migrations() -> Vec<sqlx::migrate::Migration>`, returning every embedded
  Nexora up/down migration for combination with application migrations. Each call returns an
  independent list and performs no database I/O. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Changed

- `Server::initialize(&settings, &pool, setup_secret)` now only assembles Account, OIDC, ZITADEL,
  and routers. The host must migrate first. `Server::migrate` and `nexora::server::migrate` were
  removed. — Owner: [@Jixiangup](https://github.com/Jixiangup)
- Framework and application migrations must be checked for cross-source version collisions,
  combined into one SQLx `Migrator`, and run once. The combined global version order controls
  execution; running a framework Migrator followed by an application Migrator is unsupported. —
  Owner: [@Jixiangup](https://github.com/Jixiangup)
- The Account workspace scaffold, server example, repository rules, generated-project rules, and
  related Skills now migrate explicitly before Server initialization. The bilingual server guides
  document the same ownership boundary. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- GitHub-tag installation now selects the `nexora` CLI package explicitly when the repository
  contains multiple binary packages. — Owner:
  [@Jixiangup](https://github.com/Jixiangup)

## Compatibility and breaking changes

### Previous pattern (0.1.2)

`Server::initialize` implicitly ran Nexora's own Migrator first. Hosts could also call
`Server::migrate` directly. That pattern knew only framework migrations and could not safely
compose application migrations:

```rust
let mut server = nexora::Server::new();
server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
```

### New pattern (0.2.0)

The application collects both sources first. `reject_cross_source_version_conflicts` below denotes
a fail-closed helper owned by the application's migration crate. It must reject the same numeric
version across sources even when SQL text or checksums happen to match. Build and run one Migrator,
then initialize Nexora only after migration succeeds:

```rust
let framework_migrations = nexora::server::migrations();
let application_migrations = application_migrations();

reject_cross_source_version_conflicts(
    framework_migrations.as_slice(),
    application_migrations.as_slice(),
)?;

let migrations = framework_migrations
    .into_iter()
    .chain(application_migrations)
    .collect();
sqlx::migrate::Migrator::with_migrations(migrations)
    .run(&pool)
    .await?;

let mut server = nexora::Server::new();
server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
```

Application and Nexora versions now share one global numeric namespace. SQLx runs the combined list
in version order. Do not use source insertion order to express dependencies, and do not set
`ignore_missing = true` to hide unknown history. The application migration crate must be the only
migration composition root.

### Database migration

Nexora 0.2.0 adds or rewrites no DDL, so the version upgrade itself requires no data movement. The
unified Migrator recognizes and skips successfully applied Nexora versions and checksums already in
`_sqlx_migrations`. Do not clear, rewrite, or fabricate those rows.

If application migrations are already recorded in the same database, verify before startup that
their numeric versions do not collide with Nexora versions. On collision, stop and re-plan only
unshared application migrations with globally unique versions. Never rename, reorder, or modify the
checksum of migrations that have entered a shared environment.

## Upgrade from 0.1.2

1. Stop writes and back up PostgreSQL, `_sqlx_migrations`, and production configuration.
2. Pin Nexora to `v0.2.0` and update the lockfile:

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.2.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

3. Remove calls to `Server::migrate` or `nexora::server::migrate`. Make the application migration
   crate collect `nexora::server::migrations()` and application migrations and reject duplicate
   numeric versions across the two sources before database access.
4. Construct one `Migrator` from the combined list and run it once on the application's one
   `PgPool`. Only after success should the host call `server.initialize(...)`, compose routers, and
   bind its listener.
5. Verify `_sqlx_migrations`, Account login, user/role/permission management, and application
   routers against a pre-production database before restoring production traffic.
6. Reinstall the 0.2.0 CLI:

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.2.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## Rollback notes

- If the unified Migrator has recorded no application migration, the database still contains only
  Nexora history known to 0.1.2. A code rollback must still restore the old API calls and lockfile.
- If application migrations have been recorded, changing only the dependency tag to 0.1.2 is not
  safe. The 0.1.2 framework-only Migrator cannot see those application versions and returns
  `VersionMissing` with SQLx's default `ignore_missing = false`; the old startup path cannot resume
  directly.
- Prefer restoring the pre-upgrade database backup. For an in-place rollback, keep using the 0.2.0
  unified migration entry point, verify application down migrations and data compatibility in
  pre-production, and roll application history back until only framework history known to 0.1.2
  remains before deploying the old binary. If no verified down path exists, do not manually delete
  `_sqlx_migrations` rows; restore the backup or fix forward.
