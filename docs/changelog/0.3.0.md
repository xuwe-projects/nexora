# Nexora 0.3.0

发布日期：2026-07-18

本版本补齐 Account 与宿主业务 Router 的组合边界，并把桌面端默认用户、角色和权限页面从
只读展示扩展为可完成日常管理的交互。应用现在可以从已经初始化的 `Server` 取得 Account
句柄，把它放进自定义 Axum State，并直接复用 Nexora 的认证与权限 extractor；用户开通也
可以在同一事务中建立初始角色关系。

## Added

- 新增 `Server::account()`，在 `Server::initialize` 成功后返回可克隆的 `Account` 句柄。
  句柄复用初始化时传入的唯一 `PgPool`，不会创建第二个连接池；初始化前返回 `None`。—
  处理人：[@Jixiangup](https://github.com/Jixiangup)
- `AuthenticatedUser` 与 `Authorized<P>` 现在可用于任意满足
  `Account: axum::extract::FromRef<AppState>` 的自定义 Axum State。宿主不再需要复制 token
  校验、本地用户状态检查或权限合并逻辑。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 新增 pool-first `create_user_with_roles` 和 `Account::provision_user_with_roles`。它们使用
  已确认的 `ExternalIdentity` 创建本地用户，自动保留内置 `member` 角色，并以本地
  `granted_by` 用户 ID 原子写入初始业务角色；任一角色或授权人无效时整笔事务回滚。—
  处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- `ProvisionUserRequest` 新增 `role_ids`。默认 `POST /users` 可以在开通用户时直接建立初始
  角色关系；从 0.3.0 起，空集合也会授予不带权限的内置 `member` 角色，因此有效权限仍为空，
  但 `user_roles` 关系会相较 0.2.0 增加。非空初始角色同时要求
  `users:provision` 与 `users:roles.write`，不能只凭用户开通权限分配角色。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 默认 `/users` 页面现在支持列表刷新、开通已确认的 OIDC 身份、创建时选择初始角色、启用或
  停用普通用户，以及完整替换普通用户的直接角色集合。超级管理员保护和最后一个有效管理员
  约束仍由 Account 服务端统一执行。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 默认 `/roles` 页面现在支持创建自定义角色并选择初始权限、更新角色名称与说明、替换完整
  权限集合和删除自定义角色。创建角色及其初始权限继续统一使用 `roles:write`；内置角色保持
  不可修改。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 默认管理页面按 `users:provision`、`users:status.write`、`users:roles.write`、
  `roles:read`、`roles:write` 与 `permissions:read` 控制可见操作和说明，不在桌面端硬编码
  角色名称。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## 兼容性与破坏性变更

### 自定义 Axum State

现有只合并 `server.routers()` 的 0.2.0 应用可以保持原样。只有业务 Router 需要直接复用
Account 认证或权限 extractor 时，才需要把初始化后的 Account 句柄加入自己的 State，并
实现 `FromRef`：

```rust
use axum::extract::FromRef;
use nexora::server::Account;
use sqlx::PgPool;

#[derive(Clone)]
struct AppState {
    pool: PgPool,
    account: Account,
}

impl FromRef<AppState> for Account {
    fn from_ref(state: &AppState) -> Self {
        state.account.clone()
    }
}

server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
let account = server.account().expect("Server 已完成初始化");
let state = AppState {
    pool: pool.clone(),
    account,
};

let app = axum::Router::new()
    .merge(server.routers())
    .merge(application_routes())
    .with_state(state);
```

业务 handler 随后可以直接提取 `AuthenticatedUser`，或通过实现 `RequiredPermission` 使用
`Authorized<P>`。`authorization.profile().user.id` 是当前 Nexora 本地用户 ID，可作为业务
审计字段；extractor 不向业务代码暴露 bearer token。

### `ProvisionUserRequest` 源码与 JSON 兼容性

- Rust 代码如果使用 struct literal 构造 `ProvisionUserRequest`，升级后必须显式增加
  `role_ids: Vec::new()` 或实际角色 ID。这是源码迁移，不能因为 JSON 向后兼容而省略。
- `role_ids` 反序列化时有空集合默认值，因此 0.2.0 JSON 客户端省略该字段的请求仍可被
  0.3.0 服务端接受。0.3.0 服务端会为这类请求补授不带权限的内置 `member` 角色；有效权限
  仍为空，但角色关系快照不再与 0.2.0 完全相同。
- 0.3.0 客户端在 `role_ids` 为空时省略该字段，因此仍可向 0.2.0 服务端发送原有用户开通
  请求。非空 `role_ids` 是 0.3.0 能力，客户端与服务端都应升级后再使用。
- 非空 `role_ids` 需要当前操作者同时拥有 `users:provision` 和 `users:roles.write`；创建角色
  时提交非空 `permission_ids` 仍只要求既有的 `roles:write`。

### 数据库迁移

0.3.0 没有新增、删除或改写 Nexora DDL，也没有新增迁移版本。既有 0.2.0 数据库和
`_sqlx_migrations` 历史无需手工数据迁移；应用继续使用 0.2.0 引入的唯一组合 Migrator。

## 从 0.2.0 升级

1. 把 Nexora 依赖固定到 `v0.3.0` 并更新锁文件：

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.3.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

2. 为代码中的 `ProvisionUserRequest` struct literal 增加 `role_ids`。不需要创建时分配角色的
   调用方使用 `Vec::new()`。
3. 业务 Router 需要 Account extractor 时，在 `server.initialize(...)` 成功后读取
   `server.account()`，把句柄放入最终 State，并实现 `FromRef<AppState> for Account`。只使用
   Nexora 默认 Router 的应用不需要这一步。
4. 若使用创建时角色分配，确认操作者同时拥有 `users:provision` 和
   `users:roles.write`；角色创建及初始权限继续授予 `roles:write`。
5. 验证旧式空角色用户开通、带初始角色开通、用户启停、用户角色替换、角色增删改和权限替换。
6. 重新安装 0.3.0 CLI：

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.3.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## 回滚注意事项

- 本版本没有数据库结构变化，因此数据库本身不需要 down migration；仍应先在预发布环境验证
  应用代码和依赖锁文件的回退。
- 回退到 0.2.0 前，删除对 `Server::account`、通用 Account extractor 和
  `create_user_with_roles` 的调用，并恢复 0.2.0 的 `ProvisionUserRequest` struct literal。
- 0.2.0 服务端不认识非空 `role_ids`。回滚期间必须停用创建时角色分配，或改为先按旧请求
  开通用户，再通过既有用户角色替换接口完成授权。
