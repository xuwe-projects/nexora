# Nexora 0.2.0

发布日期：2026-07-17

本版本把 Account 的可信宿主管理能力和数据库迁移组合权交给应用。宿主现在可以直接使用
自己的唯一 `PgPool` 创建用户、权限和角色并维护关联关系；Nexora 只导出框架迁移清单，
`Server::initialize` 不再隐式修改数据库。由于删除旧迁移入口并改变初始化前置条件，本版本
包含破坏性变更。

## Added

- 在 `nexora::server` 增加五个 pool-first Account API：`create_permissions`、
  `create_role`、`replace_role_permissions`、`create_user` 和 `replace_user_roles`。这些函数
  复用 Account 既有校验、Store 与事务，不创建第二个连接池；调用方必须处于可信宿主边界，
  并自行完成当前操作者认证授权。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- `create_permissions` 幂等创建或更新权限目录；`create_role` 在同一事务中写入自定义角色及
  初始完整权限集合；两个 `replace_*` API 原子替换完整关联集合，而不是增量追加；
  `create_user` 只开通已经由宿主确认的外部身份，不引入本地密码。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 增加 `nexora::server::migrations() -> Vec<sqlx::migrate::Migration>`，返回 Nexora 维护的
  全部嵌入式 up/down 迁移，供应用与自己的迁移清单组合。每次调用返回独立列表，函数不会
  访问数据库。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- `Server::initialize(&settings, &pool, setup_secret)` 现在只装配 Account、OIDC、ZITADEL 与
  Router。宿主必须在调用前执行统一迁移；删除 `Server::migrate` 和
  `nexora::server::migrate`。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 框架迁移与应用迁移必须先检查跨来源版本冲突，再合并进唯一 SQLx `Migrator` 并执行一次。
  迁移按合并后的全局版本顺序执行，不能先跑框架 Migrator 再跑应用 Migrator。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- Account workspace 脚手架、Server 示例、仓库规则、生成项目规则和相关 Skills 已改为显式
  统一迁移后再初始化 Server；中英文服务端文档同步记录新的资源所有权边界。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- Git 仓库同时包含多个二进制 package 时，GitHub tag 安装命令现在通过位置参数 `nexora`
  明确选择 CLI package。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## 兼容性与破坏性变更

### 旧写法（0.1.2）

`Server::initialize` 会先隐式运行 Nexora 自己的 Migrator；宿主也可以单独调用
`Server::migrate`。这套写法只知道框架迁移，无法安全组合应用迁移：

```rust
let mut server = nexora::Server::new();
server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
```

### 新写法（0.2.0）

应用先收集框架和业务迁移。`reject_cross_source_version_conflicts` 是应用迁移 crate 的
fail-closed 检查：只要两个来源使用相同数字版本就应返回错误，即使 SQL 文本或校验和碰巧
相同也不能继续。检查通过后构造一个 Migrator，成功完成后才初始化 Nexora：

```rust
let framework_migrations = nexora::server::migrations();
let application_migrations = application_migrations();

reject_cross_source_version_conflicts(
    framework_migrations.as_slice(),
    application_migrations.as_slice(),
)?;

let migrations = framework_migrations
    .into_iter()
    .chain(application_migrations)
    .collect();
sqlx::migrate::Migrator::with_migrations(migrations)
    .run(&pool)
    .await?;

let mut server = nexora::Server::new();
server
    .initialize(&settings, &pool, settings.setup.secret()?)
    .await?;
```

应用迁移版本必须与 Nexora 的版本形成同一个全局命名空间。组合列表交给 SQLx 后按版本顺序
执行；不要依赖“先放框架、后放应用”的输入顺序来表达依赖，也不要设置
`ignore_missing = true` 绕过未知历史。应用 migration crate 应是唯一迁移 composition root。

### 数据库迁移

0.2.0 没有新增或改写 Nexora DDL，因此从 0.1.2 升级本身不需要数据搬运。既有
`_sqlx_migrations` 中成功应用的 Nexora 版本和校验和会被统一 Migrator 识别并跳过；不要
清空、重写或手工伪造这些记录。

如果应用此前已经在同一个数据库中记录了自己的迁移，升级前必须先确认其数字版本没有与
Nexora 迁移冲突。发生冲突时停止启动并通过新的、全局唯一版本重新规划尚未共享的应用
迁移；已经进入共享环境的迁移不得改名、重排或修改校验和。

## 从 0.1.2 升级

1. 停止写入服务，备份 PostgreSQL 数据库、`_sqlx_migrations` 和生产配置。
2. 把 Nexora 依赖固定到 `v0.2.0` 并更新锁文件：

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.2.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

3. 删除对 `Server::migrate` 或 `nexora::server::migrate` 的调用。让应用 migration crate
   收集 `nexora::server::migrations()` 和业务迁移，并在访问数据库前拒绝跨来源数字版本
   冲突。
4. 使用合并后的列表构造唯一 `Migrator`，在应用唯一 `PgPool` 上执行一次。迁移成功后再
   调用 `server.initialize(...)`，最后组合 Router 和绑定监听器。
5. 在预发布数据库上核对 `_sqlx_migrations`、Account 登录、用户/角色/权限管理和业务
   Router，再恢复生产流量。
6. 重新安装 0.2.0 CLI：

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.2.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## 回滚注意事项

- 如果统一 Migrator 尚未记录任何应用迁移，数据库层仍只有 0.1.2 已认识的 Nexora 历史；
  回退代码时仍需恢复旧 API 调用和依赖锁文件。
- 如果统一 Migrator 已经记录应用迁移，不能只把依赖 tag 改回 0.1.2。0.1.2 的
  framework-only Migrator 看不到这些应用版本，在 SQLx 默认 `ignore_missing = false` 下会
  返回 `VersionMissing`，旧启动方式无法直接恢复。
- 优先恢复升级前的数据库备份。若必须原地回滚，先继续使用 0.2.0 的统一迁移入口，在预发布
  环境验证应用 down migration 和数据兼容性，并把应用迁移安全回退到仅剩 0.1.2 能识别的
  框架历史后再部署旧二进制。没有可验证 down 路径时不要手工删除 `_sqlx_migrations` 行，
  应恢复备份或继续前向修复。
