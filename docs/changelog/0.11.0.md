# Nexora 0.11.0

发布日期：2026-07-19

本版本继续处于 `0.x` beta 阶段，不进入 `1.x` 稳定通道；版本号使用标准 SemVer `0.11.0`，
不附加 beta 后缀。由于本次新增服务端公开 API，并且 `VerifiedIdentity` 公开结构体字段发生
变化，在 `0.x` 阶段按 SemVer 升级 minor 版本。

## Added

- 新增 `nexora::server::ZitadelProvisioningClient`，用于动态创建/查找 ZITADEL Organization、
  为 portal project 创建或更新 Project Grant、创建/停用/删除目标 Organization 下的人类用户，
  以及创建/更新/删除 portal project authorization。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 新增 `Server::zitadel_provisioning_client()` 与
  `nexora::server::provisioning_client(&settings)`，复用 Account 服务端现有 issuer、PAT、
  TLS、loopback 开发规则和敏感日志遮蔽。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增独立的 OIDC resource server 认证入口：
  `OidcAccessTokenVerifier::discover_many`、`OidcResourceServer`、`BearerAccessToken` 和
  `VerifiedBearerIdentity`，用于 customer portal 或 public API 验证独立 audience，而不绑定
  Nexora Account 本地用户表或 RBAC。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- `VerifiedIdentity` 新增可选的 `organization: Option<VerifiedOrganizationContext>`，从
  ZITADEL access token 中保留 Organization id、name 与 primary domain，方便业务模块映射
  customer/portal tenant。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- `ZitadelUserDirectory` 复用新的 ZITADEL gRPC channel/PAT 构造边界，避免 Account 用户目录和
  portal provisioning 维护两套 TLS、loopback 与敏感日志规则。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 本地 ZITADEL proto 最小契约补齐 Organization、Project Grant、Authorization 与 User
  deactivate 所需消息和 RPC，供框架级 provisioning client 使用。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- 认证错误、配置错误和 gRPC client Debug 输出继续避免回显 access token、PAT 或 Provider
  内部细节；新增测试覆盖 portal verifier 错误映射与 PAT Debug 遮蔽。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## Issue 与 PR

- Issue：无。本版本来自 iMES 客户管理与客户门户开通对 Nexora 服务端框架能力的需求。
- PR：无；按发布要求由默认分支直接提交并发布。

## 兼容性与破坏性变更

- 破坏性变更：`VerifiedIdentity` 公开结构体新增 `organization` 字段。下游测试或自定义
  `AccessTokenVerifier` 如果使用结构体字面量构造 `VerifiedIdentity`，需要补充：

```rust
organization: None
```

- 功能保持兼容：下游仍只需要使用 `nexora` 的 `server` / `desktop` feature；不需要也不应该直接
  依赖内部 `zitadel` feature。
- 默认 Account `/users`、`/roles`、`AuthenticatedUser` 与 `Authorized<P>` 行为保持兼容；
  customer portal 用户不会混入默认内部 Account 管理页。
- 无数据库迁移、OpenAPI 路由、默认 Account 配置或桌面 UI 行为变更。回滚到 `v0.10.0` 需要移除
  对新增 provisioning/resource-server API 的调用，并回退 `VerifiedIdentity` 字面量字段。

## 升级

1. 把 Nexora tag 从 `v0.10.0` 更新为 `v0.11.0` 并重新锁定依赖。
2. 搜索 `VerifiedIdentity {`，为结构体字面量补齐 `organization: None` 或携带已验证的
   `VerifiedOrganizationContext`。
3. iMES 或其他宿主要开通客户 portal 时，在 `Server::initialize` 后取
   `server.zitadel_provisioning_client()`，或使用 `nexora::server::provisioning_client(&settings)?`。
4. 客户创建流程建议先查找或创建 ZITADEL Organization，并确保平台服务账号或指定管理员在
   `administrator_user_ids` 中；随后调用 `ensure_project_roles`、`ensure_project_grant`、
   `create_human_user` 和 `ensure_authorization`。业务数据库写入失败时用 `delete_user` 做补偿。
5. portal/openapi 后端使用 `OidcAccessTokenVerifier::discover_many(issuer, audiences)` 构造
   `OidcResourceServer` 并放入 Axum State，通过 `VerifiedBearerIdentity` 读取
   `identity.organization.id` 映射业务租户；不要复用 Account 的本地用户表/RBAC extractor。
6. 不需要同步新的脚手架 Skill 或运行数据库迁移。本版本未修改默认生成项目结构。
