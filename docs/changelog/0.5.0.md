# Nexora 0.5.0

发布日期：2026-07-18

本版本把导航目录、内容区表单对话框、Sidebar 组合边界和级联选择器收进 Nexora 正式公共
API，并完成默认用户/角色管理与 ZITADEL 的服务端闭环。Feature 父目录模型发生不兼容变化，
因此在 `0.y.z` 阶段严格按 Semantic Versioning 递增 MINOR 到 0.5.0。

## Added

- 新增 `#[derive(nexora::NavigationGroup)]`：纯目录只有稳定 ID、标题、section、icon、order 和
  可选父目录，没有 path、页面 factory、Window、Tab 或最近页面记录。支持任意层级、稳定排序、
  活动叶子祖先自动展开，以及只显示不可点击目录标题的面包屑。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 新增 `nexora::desktop::{FormDialog, FormDialogState, FormFieldDraft}`：统一标题、描述、可滚动
  内容区与默认取消/提交操作；遮罩只覆盖当前 Feature Panel，默认取消会列出未保存字段和草稿，
  同时允许调用方覆盖取消与提交逻辑。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增 `nexora::desktop::{Cascader, CascaderState, CascaderOption, CascaderSelection}`。组件组合
  gpui-component 的 Popover、Input、Button、Icon 与滚动能力，支持稳定值路径、搜索、禁用、
  清空、自定义分隔符和非叶子变更。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增 `SidebarRegion`，由调用方提供稳定唯一 Element ID，让品牌、应用 Context、工厂选择器
  和账户区域分别管理 hover 与点击。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增迁移 7，回填系统管理员角色缺失的全部权限；以后宿主每次注册权限时也会在同一事务中
  自动授予系统管理员。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增无锁脚手架 consumer `cargo check` 发布门禁，以及 `nexora lint` 的
  `nexora::gpui_revision_mismatch` 诊断。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- `Feature` 只表示实际页面。派生属性 `parent` 已删除，叶子页面使用
  `group = "navigation-group-id"`；NavigationGroup 的 `parent` 只能引用同一 section 的另一个
  NavigationGroup。注册阶段会明确拒绝重复 ID、未知引用、自引用、循环和跨 section 引用。
  — 处理人：[@Jixiangup](https://github.com/Jixiangup)
- Shell 继续托管 Sidebar Header/Footer 的宽度、间距、内边距和分隔线，但不再向自定义插槽
  注入 hover、selected、圆角、cursor 或点击语义。自定义 Header 现在追加在默认品牌之后，
  不会再把 Application Logo 完整替换掉；默认账户 Footer 自己显式声明 hover。
  — 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 默认用户 DataTable 改为内容高度驱动并设置内部最大滚动高度，保留无限加载，同时支持列顺序
  移动和列宽调整。用户列显示头像，登录用户名和邮箱使用自然宽度，状态使用紧凑 Tag，内部
  `identity_id` 不再显示；卡片使用与工作区不同的 `group_box` 内容面。
  — 处理人：[@Jixiangup](https://github.com/Jixiangup)
- `POST /users` 不再接收客户端提供的裸 `identity_id`。请求改为 username、given_name、
  family_name、email、可选 display_name 与 role_ids；服务端通过 ZITADEL UserService v2 gRPC
  创建人类用户，再用返回的 identity ID 原子绑定本地用户和角色。`GET /me` 会从 ZITADEL
  同步用户名、邮箱、展示名与头像。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 默认创建用户、管理用户角色、创建角色和编辑角色统一迁入 Panel-scoped FormDialog；角色列表
  改为紧凑卡片行，系统管理员显示独立标识。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 服务端 OIDC 配置新增必填 `organization_id`，用于确定 ZITADEL 创建用户的 Organization。
  — 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 项目与脚手架统一锁定经过完整验证的依赖组合：
  `gpui-component`/assets revision `031555662e99a1b5a549990b47f246d475b8288a`，以及该版本
  lockfile 对应的 Zed/GPUI revision `1a246efd7e1b83ab568ec5e3e6c1a43a42e1abba`。模板同时
  固定 `gpui` 并替换 gpui/gpui_macros 的未固定传递 source，避免下游无 lockfile 时生成两套
  不兼容类型。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## 破坏性变更与迁移

### 1. 把父 Feature 拆成 NavigationGroup

旧写法已经在编译期明确报错：

```rust
#[derive(Default, nexora::Feature)]
#[nexora(title = "车间", path = "/workshops", parent = "resources")]
struct WorkshopsFeature;
```

改为纯目录和叶子页面：

```rust
#[derive(nexora::NavigationGroup)]
#[nexora(id = "resources", title = "生产建模", section = "资料中心")]
struct ResourcesGroup;

#[derive(Default, nexora::Feature)]
#[nexora(
    title = "车间",
    path = "/workshops",
    section = "资料中心",
    group = "resources"
)]
struct WorkshopsFeature;
```

不要保留旧父 Feature 的占位页或业务侧第二套路由表。目录与叶子必须在同一 section。

### 2. 更新用户创建请求与服务端配置

在 `[oidc]` 增加 `organization_id`，并确保服务账号 PAT 有权在该 Organization 调用
UserService v2。客户端创建用户改发：

```json
{
  "username": "lin.chen",
  "given_name": "Lin",
  "family_name": "Chen",
  "email": "lin@example.com",
  "display_name": "Lin Chen",
  "role_ids": []
}
```

删除旧请求中的 `identity_id` 与 `avatar_url`。升级顺序是：备份数据库；更新配置；运行包含
迁移 7 的唯一合并 Migrator；部署服务端；重锁桌面应用依赖；再部署桌面端。

### 3. 更新 Sidebar 自定义区域

自定义 Header/Footer 不再继承 Shell hover。把需要交互的 Context 包进带稳定 ID 的
`SidebarRegion` 并显式添加 hover/click；没有动作的品牌可以完全不设置 hover。移除为抵消旧
父 hover 而存在的负 margin 或背景覆盖。

### 4. 重锁 GPUI 依赖

删除下游错误解析的 `Cargo.lock`，让根 manifest 中的固定 revision 重新解析；直接使用
gpui/gpui-component 的应用必须与生成模板保持完全相同的 revision 和 `[replace]` 条目。
运行 `nexora lint --workspace . --deny-warnings` 和干净 lockfile 的 `cargo check`。

## 回滚

- 已发布 tag 不得移动。若必须回滚代码，先恢复升级前的数据库备份；迁移 7 的 down 是安全
  no-op，因为无法区分迁移前已有授权与回填授权，保留管理员权限比删除合法权限更安全。
- 回滚到 0.4.1 前必须恢复旧 Feature parent 声明、旧 Sidebar 组合方式和旧 `POST /users`
  契约；已经在 ZITADEL 创建的新用户不会由代码回滚自动删除。
- 依赖 revision 应与代码版本一起回滚，不能只移动 gpui-component 而保留另一组 GPUI。
