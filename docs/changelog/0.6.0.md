# Nexora 0.6.0

发布日期：2026-07-18

本 early-alpha 版本调整桌面 Shell 的 `SidebarHeader` 插槽语义：应用声明
`#[derive(nexora::SidebarHeader)]` 后会接管整个 Header 内容，不再与框架默认品牌区域同时
渲染。这个行为影响已经自定义 Sidebar Header 的应用，因此在 `0.y.z` 阶段按 Semantic
Versioning 递增 MINOR。

## Changed

- 自定义 `SidebarHeader` 现在替换默认品牌区域。没有自定义 Header 时，Shell 仍显示
  `ApplicationOptions` 中的应用名称、Logo 和 subtitle；存在自定义 Header 时，Shell 只保留
  Header 的结构边界和下方分隔线，由应用实现完整内容。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 更新 Sidebar Shell 源码契约测试，锁定自定义 Header 必须优先于默认品牌 fallback，并防止
  未来重新把自定义 Header 追加到默认品牌之后。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 修正文档中“自定义 Header 追加在默认品牌之后”的旧描述，明确需要品牌与应用 Context 并存时
  应在自定义 Header 内使用稳定 ID 的 `SidebarRegion` 自行组合。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## Issue 与 PR

- Issue：无。本版本来自 iMES 桌面接入时对 Sidebar Header 插槽替换语义的明确要求。
- PR：无；按发布要求由默认分支直接提交并发布。

## 兼容性与破坏性变更

- 破坏性变更：已经声明自定义 `#[derive(nexora::SidebarHeader)]` 的应用将不再自动显示默认
  Logo、应用名和 subtitle。
- 未声明自定义 Header 的应用无行为变化，仍继续显示框架默认品牌区域。
- `SidebarFooter`、Feature、NavigationGroup、Window、SettingsWindow 和 LoginFeature 的声明
  与生命周期不变。
- HTTP API、OpenAPI 路径、ZITADEL 集成、数据库 schema、迁移历史、`gpui` revision 与
  `gpui-component` revision 均未变化。

## 升级

1. 把 Nexora tag 从 `v0.5.2` 更新为 `v0.6.0` 并重新锁定依赖。
2. 如果应用声明了自定义 `SidebarHeader`，检查是否仍需要展示 Logo、应用名或 subtitle。
3. 需要保留品牌信息时，在自定义 Header 的 `Render` 实现中显式渲染这些内容；需要多个独立
   命中区域时，使用稳定 ID 的 `nexora::desktop::SidebarRegion` 分别组合品牌、租户或工厂选择器。
4. 不需要运行数据库迁移或修改服务端配置。回滚到 `v0.5.2` 会恢复“默认品牌 + 自定义 Header”
   同时渲染的旧行为。
