# Nexora 0.4.0

发布日期：2026-07-18

本版本重做默认用户管理体验：用户列表改为卡片化 `DataTable`，支持虚拟化无限滚动、头像、
登录用户名和语义化状态；创建用户改用高度受限的标准 `Dialog`，标题、可滚动内容区和底部
操作区保持清晰分层。同时 Account 契约和数据库新增可选登录用户名，并继续用稳定
`identity_id` 作为认证绑定主键。

## Added

- `ExternalIdentity`、`ProvisionUserRequest`、`User` 与 `UserResponse` 新增可选 `username`。
  OIDC 登录会从 `preferred_username` 同步该资料，管理员创建用户时也可以显式绑定登录
  用户名；旧 JSON 请求省略该字段时继续有效。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 新增 SQLx 迁移 `0006_account_add_username`，在 `account.users` 增加允许为空、最长 200 个
  字符的 `username` 列及完整数据库注释。既有用户保留为未绑定状态，不执行不可靠的自动
  推断。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 桌面运行时内嵌 `rotate-ccw` 图标资源，默认用户页面刷新按钮使用该图标，并继续通过统一
  GPUI AssetSource 分发。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增中英文 `nexora::server` 完整参考：HTTP 文档逐接口记录路由来源、权限、path/query/form/
  JSON 入参、响应模型、稳定错误码和示例；Rust 文档记录 `Server` 生命周期、Account facade、
  pool-first 函数、extractor 与 Setup 扩展点。OpenAPI 3.1 同步覆盖默认 Setup SSR 表单与
  Account 契约。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- 默认 `/users` 表格改用 `gpui-component::DataTable`，以稳定用户 ID 渲染虚拟行，并利用
  `TableDelegate::load_more` 在接近底部时继续请求下一页；手工 Pagination 已移除。—
  处理人：[@Jixiangup](https://github.com/Jixiangup)
- 用户列显示头像、显示名称、本地用户 ID 和超级管理员标记；单独展示登录用户名，隐藏不适合
  日常管理界面直接暴露的 `identity_id`。访问状态改用成功/警告 `Tag`，不再只显示纯文字。—
  处理人：[@Jixiangup](https://github.com/Jixiangup)
- DataTable 使用共享 `Card` 承载，卡片与表格读取 `group_box.background` /
  `table.background`，与桌面页面背景形成明暗层级，并保留边框、圆角和主题阴影。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- 主操作统一为“创建用户”。创建流程改用 `gpui-component` 标准 `Dialog`：固定 Title 和
  Footer，内容区在超过 640px 最大高度时仅沿 Y 轴滚动，表单包含登录用户名、Identity ID、
  显示名称、邮箱、头像 URL 与初始角色。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)
- `publish-nexora-release` Skill 与脚手架模板新增严格 SemVer 门禁：按 breaking、feature、
  fix 的最高影响选择版本，拒绝不递增、降级或级别不匹配的目标版本。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## Fixed

- 刷新数据现在始终从第一页原子替换现有行；无限加载只追加尚未出现的用户 ID，降低 offset
  分页期间重复行对界面的影响。状态修改和角色编辑会局部通知 DataTable 刷新。— 处理人：
  [@Jixiangup](https://github.com/Jixiangup)

## 兼容性与破坏性变更

- HTTP JSON 保持向后兼容：`ProvisionUserRequest.username` 有缺省值，`UserResponse.username`
  也允许旧服务端响应省略。`identity_id` 的认证与唯一性语义没有变化。
- Rust 源码存在 struct-literal 迁移：直接构造 `ExternalIdentity`、`ProvisionUserRequest`、
  `User` 或 `UserResponse` 的调用方需要增加 `username: None` 或实际值。Nexora 仍处于
  `0.y.z` 初始开发阶段，因此按严格 SemVer 递增 MINOR 到 0.4.0。
- 数据库新增可空列和约束，不删除或重写既有数据。应用代码部署前必须先使用合并后的唯一
  SQLx Migrator 应用迁移 6，否则 0.4.0 查询会因缺少 `username` 列失败。

## 从 0.3.1 升级

1. 把 Nexora 依赖固定到 `v0.4.0` 并更新锁文件：

   ```toml
   nexora = { git = "https://github.com/xuwe-projects/nexora", tag = "v0.4.0", default-features = false }
   ```

   ```bash
   cargo update -p nexora
   ```

2. 在发布 0.4.0 应用代码前，把 `nexora::server::migrations()` 与应用迁移合并，并使用现有
   唯一 Migrator 应用迁移 6。
3. 为 Rust struct literal 增加 `username`。没有可信用户名来源时使用 `None`；不要把
   `identity_id` 猜成用户名。
4. 若使用自定义 Account API 客户端，可开始在 `POST /users` 中发送可选 `username`，并在
   用户响应中读取该字段。旧 JSON 客户端不需要立即修改。
5. 验证创建用户、头像回退、状态 Tag、角色管理、刷新和滚动到底部后的连续加载。
6. 重新安装 0.4.0 CLI：

   ```bash
   cargo install --git https://github.com/xuwe-projects/nexora --tag v0.4.0 nexora --locked --force --no-default-features --features cli --bin nexora
   ```

## 回滚注意事项

- 回退到 0.3.1 代码时可以保留新增的可空 `username` 列；0.3.1 的显式字段查询会忽略它，
  这是风险最低的回滚方式。不要手工删除 `_sqlx_migrations` 记录。
- 如果必须执行 down migration，先备份已经绑定的用户名，再删除 0.4.0 代码对该字段的读写；
  down migration 会永久删除 `account.users.username` 中的数据。
- 回退源码前，从 `ExternalIdentity`、`ProvisionUserRequest`、`User` 和 `UserResponse` 的 struct
  literal 中移除 0.4.0 新字段。
