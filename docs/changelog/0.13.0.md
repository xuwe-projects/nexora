# Nexora 0.13.0

发布日期：2026-07-20

本版本继续处于 `0.x` beta 阶段，不进入 `1.x` 稳定通道；版本号使用标准 SemVer `0.13.0`。
本次发布为 iMES 增加桌面导航可见权限和 Account 权限蕴含保存能力。

## Added

- Feature 静态元数据新增客户端可见权限，派生宏支持 `visible_permissions(any = ["employees:read"])`；
  手写 Feature 元数据可通过 `FeatureVisiblePermissions`、`with_visible_permissions` 或
  `with_visible_permissions_any` 使用同一能力。该能力只影响桌面导航和路由可见性，不替代服务端
  `Authorized<P>` 授权。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 桌面主导航会先隐藏当前登录用户不可见的 Feature，再隐藏没有任何可见子项的 NavigationGroup；
  未登录用户看不到受限 Feature，超级管理员始终可见。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- Account 权限目录新增 `PermissionCatalogDefinition` 和 `PermissionDefinition::with_implies`，用于声明权限蕴含关系。
  旧的 `PermissionDefinition` 和 `create_permissions`/`register_permissions` 调用保持兼容。— 处理人：[@Jixiangup](https://github.com/Jixiangup)

## Changed

- 创建角色或替换角色权限时，Account 会按已注册的蕴含关系递归展开权限、去重并稳定排序，然后把最终权限集保存到
  `account.role_permissions`。例如授予 `employees:write` 且其声明蕴含 `employees:read` 时，数据库会同时保存
  `employees:write` 和 `employees:read`；`/me` 只读取最终集合，不在运行时再次推导。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 新增迁移 `0008_account_permission_implications`，保存权限蕴含关系，并为内置 Account 写权限补入对应读权限关系。
  迁移会回填已有角色权限，确保升级后的 `role_permissions` 立即保持最终集合语义。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 默认 Account 用户与角色管理 Feature 增加可见权限元数据：`users:read` 与 `roles:read`。— 处理人：[@Jixiangup](https://github.com/Jixiangup)
- 根 workspace 版本、内部 workspace 依赖约束、CLI 安装示例和 OpenAPI 元数据同步更新到 `0.13.0`。—
  处理人：[@Jixiangup](https://github.com/Jixiangup)

## Issue 与 PR

- Issue：无。本版本来自 iMES 桌面导航权限可见性与写权限默认蕴含读权限的集成需求。
- PR：无；按发布要求由当前分支直接提交并发布。

## 兼容性与破坏性变更

- 无需修改旧的 `PermissionDefinition` 调用；未声明 `implies` 的权限行为不变。
- 数据库新增 `account.permission_implications` 表。升级时必须执行 Nexora 迁移；回滚到 `v0.12.0` 需要回滚该表及相关角色权限回填结果。
- `visible_permissions` 仅是客户端导航/路由可见性提示；所有安全边界仍必须在服务端使用 `Authorized<P>` 或等效授权。

## 升级

1. 把 Nexora tag 从 `v0.12.0` 更新为 `v0.13.0` 并重新锁定依赖。
2. iMES Feature 可按需声明 `#[nexora(visible_permissions(any = ["employees:read"]))]`。
3. iMES 权限目录可使用 `PermissionDefinition { key: "employees:write", ... }.with_implies(["employees:read"])` 注册写读蕴含关系。
4. 执行数据库迁移，确保 `account.permission_implications` 和已有角色权限回填完成。
