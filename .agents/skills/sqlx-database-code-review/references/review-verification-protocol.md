# 代码审查验证协议

在报告任何代码审查发现之前，都必须遵循本协议。跳过这些步骤会导致误报，浪费开发者时间并削弱对审查的信任。

## 防虚构（门禁 0——在其他所有门禁之前执行）

在给出**任何**结论——标记、驳回或降低发现级别——之前，必须原样复述正在判断的确切产物；引用内容必须来自你在**本轮**读取的来源：

- 对于代码发现：提供**文件:行号**以及引用的代码，并在此刻重新读取（不要依赖本次会话早先的记忆）。
- 对于差异审查：提供正在审查的实际 **diff hunk**。

> 该产物是唯一事实来源。**绝不**根据分支名称、工作目录、周边文件或记忆推断正在审查的内容。如果你的心智模型与刚读取的来源不同，**以来源为准。** 如果没有在同一轮复述结论所针对的对象，该结论无效——先给出复述，否则不要给出结论。

设置此门禁是因为 LLM 在上下文诱导下，会自信地标记文件中并不存在的代码。它在下方的硬性门禁**之前**执行。

## 项目约束优先

本协议中的通用 Rust 模式不能覆盖 Xuwe 的仓库约束。尤其是本项目要求所有 Rust 测试放在
对应 crate 的 `tests/`，并禁止在生产源码中新增 `#[cfg(test)]` 或内联 `mod tests`。
审查测试代码时先应用 `rust-code-style`；不要照搬外部项目常见的内联测试写法。

## 硬性门禁（按顺序执行）

添加发现之前，按**顺序**完成这些门禁。仅当某个门禁显然不适用时才跳过它（例如，发现与死代码或“未使用”无关时，跳过使用情况门禁）。

1. **读取范围**——**通过：** 指明完整读取的确切文件路径，以及函数、`impl` 或 `macro_rules!` 块（不能只读取 diff hunk 或局部代码片段）。
2. **使用情况（死代码/未使用）**——**通过：** 已运行全仓库引用搜索（`rg`、IDE 引用查找或等效工具）；对于被判定为未使用的符号，说明零匹配，或列出每个匹配项并解释它为何仍能支持该发现。
3. **周边行为**——**通过：** 已检查调用方、trait impl、`#[cfg]` 或错误传播，确认这些因素是否会使该模式成为有意设计；注明一个实际检查的位置（路径 + 大致位置），或说明“搜索后没有相关项”。
4. **版本与 API**——**通过：** 已打开受审查 crate 对应的 `Cargo.toml`，并引用 `[package] edition = "..."` 行；或者说明使用默认 edition，并指出检查过的清单路径。
5. **错误与风格之别**——**通过：** 用一句话解释为什么该代码对本项目而言不正确、不健全或存在风险，而不只是另一种同样有效的风格。

## 报告前验证清单

标记**任何**问题之前，验证：

- [ ] **我已读取实际代码**——不只读取 diff 上下文，还读取了完整的函数/impl 块
- [ ] **我已搜索使用情况**——在声称“未使用”之前，搜索了全部引用
- [ ] **我已检查周边代码**——该问题可能已在其他地方处理（trait impl、错误传播）
- [ ] **我已对照当前文档验证语法**——检查 Rust edition、crate 版本和 API 变更
- [ ] **我已检查项目的 Rust edition**——Edition 2021 与 2024 会改变哪些内容是必需的、哪些是可选的（参见[版本感知审查](#版本感知审查)）
- [ ] **我已区分“错误”与“不同风格”**——两种方式都可能有效
- [ ] **我已考虑有意设计**——检查了注释、项目约定（例如 AGENTS.md 或 CLAUDE.md）和架构上下文

## 按问题类型验证

### “未使用的变量/函数”

**标记之前**，必须：
1. 搜索代码库中的所有引用（grep/find）
2. 检查它是否为 `pub`，以及 workspace 中的其他 crate 是否使用它
3. 检查它是否通过派生宏、trait 实现或条件编译（`#[cfg]`）使用
4. 验证它不是 trait 定义所要求的 trait 方法

**常见误报：**
- 方法由 trait 定义的 trait 实现
- 外部项目中仅在测试构建使用的 `#[cfg(test)]` 项；在 Xuwe 中应按上述测试位置约束单独审查
- 使用结构体字段的派生代码
- 通过 `From`/`Into` 转换使用的类型

### “缺少错误处理”

**标记之前**，必须：
1. 检查错误是否在更高层处理（调用方通过 `?` 传播错误）
2. 检查 crate 是否有封装此错误的顶层错误类型
3. 验证 `unwrap()` 不是位于测试代码中，也不是位于保证安全性的检查之后

**常见误报：**
- 测试和示例中的 `unwrap()`（预期模式）
- 验证后的 `expect("reason")`（例如，对字面量调用 `regex::Regex::new`）
- 通过 `?` 传播错误（由调用方处理）
- `let _ = tx.send(...)`——接收方可能已被丢弃时的有意设计

### “不必要的生命周期”/RPIT 捕获（Edition 2024）

**标记之前**，必须：
1. 检查项目 `Cargo.toml` 中的 Rust edition
2. 在 edition 2024 中，`-> impl Trait` 默认捕获作用域内的所有生命周期
3. 看似“不必要”的生命周期可能被隐式捕获——该代码是正确的
4. 如果作者使用 `+ use<'a>` 语法，这是精确捕获控制，并非错误

**常见误报：**
- 返回 `impl Trait` 的函数上的生命周期参数——edition 2024 会隐式捕获它们
- `+ use<'a, T>` 语法——这是新的精确捕获语法，并非错误
- 移除 edition 2024 现已自动提供的显式生命周期约束

### “缺少 unsafe 块”（Edition 2024）

**标记之前**，必须：
1. 检查代码是否位于 `unsafe fn` 内
2. 在 edition 2024 中，`unsafe_op_in_unsafe_fn` 默认为 deny——`unsafe fn` 内的 unsafe 操作**要求**显式的 `unsafe {}` 块
3. 这是 edition 所要求的行为，并非不必要的冗长写法

**常见误报：**
- `unsafe fn` 内的 `unsafe {}` 块——在 edition 2024 中**必需**，并非多余
- `unsafe extern "C" {}`——在 edition 2024 中**必需**，并非可选
- `#[unsafe(no_mangle)]` / `#[unsafe(export_name)]`——在 edition 2024 中**必需**

### “不必要的 Clone”

**标记之前**，必须：
1. 确认 clone 确实可以避免（借用检查器可能要求进行 clone）
2. 检查是否需要将该值移动到闭包/线程/任务中
3. 验证该类型不是 `Copy`（对 Copy 类型执行 clone 不会产生实际操作）
4. 检查 clone 是否位于热点路径中（测试/设置代码中的 clone 没有问题）

**常见误报：**
- `Arc::clone(&arc)`——这是 Arc 推荐使用的显式 clone 方式
- 在 `tokio::spawn` 之前执行 Clone——为满足 `'static` 约束所必需
- 在测试设置中执行 Clone——清晰度优先于性能

### “潜在竞态条件”

**标记之前**，必须：
1. 验证数据是否确实在线程/任务之间共享
2. 检查 `Mutex`、`RwLock` 或原子操作是否保护了该访问
3. 确认该类型本身尚未提供线程安全保证（例如 `Arc<Mutex<T>>`）
4. 检查所谓的“竞态”是否实际无害（例如日志记录、指标采集）

**常见误报：**
- `Arc<Mutex<T>>`——已经是线程安全的
- Tokio 通道操作——本身已同步
- `std::sync::atomic` 操作——专为并发访问而设计

### “性能问题”

**标记之前**，必须：
1. 确认代码的运行频率高到足以产生影响
2. 验证该优化会带来可测量的影响
3. 检查编译器是否已经对此进行优化（迭代器融合、内联）

**不要标记：**
- 启动/初始化代码中的内存分配
- 错误路径中的字符串格式化
- 测试代码中的 Clone
- 小型迭代器上的 `.collect()`

## 严重级别校准

### Critical（阻止合并）

**仅用于：**
- 存在不健全不变量的 `unsafe` 代码
- 通过字符串插值造成的 SQL 注入
- 释放后使用或内存安全违规
- 数据竞态（没有同步的并发修改）
- 生产代码路径因用户输入而发生 panic

### Major（应该修复）

**用于：**
- 跨模块边界时缺少错误上下文
- 在异步运行时中执行阻塞操作
- 跨 await 点持有 Mutex guard
- 多语句数据库写入缺少事务

### Minor（考虑修复）

**用于：**
- 公共项缺少文档注释
- 本可使用 `&str` 却使用了 `String` 参数
- 次优的迭代器模式
- 具有重要返回值的函数缺少 `#[must_use]`

### Informational（无需操作）

**用于：**
- 关于使用 newtype、构建器模式或类型状态的建议
- 没有实测影响的性能优化
- 建议添加 `#[non_exhaustive]`
- trait 设计的重构思路

**这些内容不会阻止审查通过。**

### 完全不要标记

- 两种方式都有效时的风格偏好（例如，针对单个变体使用 `if let` 还是 `match`）
- 没有可测量收益的优化
- 未达到生产标准的测试代码
- 生成的代码或宏输出
- 项目有意抑制的 Clippy lint

## 有效模式（不要标记）

### Rust

| 模式 | 有效原因 |
|---------|----------------|
| 测试中的 `unwrap()` | 标准测试行为——遇到意外错误时发生 panic |
| 测试设置中的 `.clone()` | 清晰度优先于性能 |
| 测试模块中的 `use super::*` | 访问父模块项的标准模式 |
| 二进制程序中的 `Box<dyn Error>` | 并非每个应用都需要自定义错误类型 |
| 结构体中的 `String` 字段 | 对于结构体字段，使用自有数据是正确的 |
| `Arc::clone(&x)` | 显式克隆 Arc 符合惯用写法并受到推荐 |
| 带原因的 `#[allow(clippy::...)]` | 有意抑制是有效的 |
| 使用 `#[expect(lint)]` 而不是 `#[allow]` | 可自清理的抑制方式（自 1.81 起稳定）——当 lint 不再触发时发出警告 |
| `unsafe fn` 内的 `unsafe {}` | edition 2024 中要求这样做（`unsafe_op_in_unsafe_fn` = deny） |
| `unsafe extern "C" {}` | edition 2024 对 extern 块的要求 |
| `#[unsafe(no_mangle)]` | edition 2024 对安全相关属性的要求 |
| `#[unsafe(export_name = "...")]` | edition 2024 对安全相关属性的要求 |
| `impl Trait` 返回值上的 `+ use<'a, T>` | edition 2024 RPIT 的精确捕获语法 |
| 将 `r#gen` 用作标识符 | `gen` 在 edition 2024 中是保留字 |
| `LazyLock` / `LazyCell` | 标准库中用于替代 `once_cell`/`lazy_static` 的类型（自 1.80 起稳定） |
| trait 定义中的 `async fn` | 静态分发可使用原生语法；需要 `dyn Trait` 时先验证对象安全性，必要时继续使用 `async-trait` 或显式 boxed future |
| `#[diagnostic::on_unimplemented]` | 自定义 trait 错误消息（自 1.78 起稳定） |

### 异步/Tokio

| 模式 | 有效原因 |
|---------|----------------|
| 对短临界区使用 `std::sync::Mutex` | Tokio 文档建议对非异步锁使用这种方式 |
| 不 join 的 `tokio::spawn` | 对带关闭信号的后台任务有效 |
| 带 `default` 分支的 `select!` | 非阻塞检查，是有意使用的模式 |
| 不使用 multi_thread 的 `#[tokio::test]` | 默认的单线程模式适用于大多数测试 |

### 测试

| 模式 | 有效原因 |
|---------|----------------|
| 测试中的 `expect()` | 可用于测试设置/断言 |
| 带 `expected` 的 `#[should_panic]` | 可用于测试 panic 行为 |
| 大型测试函数 | 集成测试可以很长 |
| 测试清理中的 `let _ = ...` | 清理错误通常无法采取有效措施 |

### 通用模式

| 模式 | 有效原因 |
|---------|----------------|
| 新代码中的 `todo!()` | 开发期间可使用的占位符 |
| 开发期间的 `#[allow(dead_code)]` | 迭代过程中很常见 |
| 一个类型具有多个 `impl` 块 | 按 trait 或关注点组织代码 |
| 复杂类型的类型别名 | 减少样板代码，提高可读性 |

## 上下文相关规则

### 所有权

**仅当**满足以下条件时，才标记不必要的 `.clone()`：
- [ ] 位于热点路径中（不是测试/设置代码）
- [ ] 可以使用借用或引用
- [ ] clone 并非为满足 `Send`/`'static` 约束所必需
- [ ] 该类型不是 `Copy`

### 错误处理

**仅当**满足以下条件时，才标记缺少错误上下文：
- [ ] 错误跨越模块边界
- [ ] 错误类型本身没有携带上下文（thiserror 消息）
- [ ] 不位于测试代码中
- [ ] 直接使用 `?` 会丢失关于哪个操作失败的有意义信息

### Unsafe 代码

**仅当**满足以下条件时，才标记 unsafe：
- [ ] 缺少安全性注释，或者注释没有解释不变量
- [ ] unsafe 块的范围超出必要范围
- [ ] 周边代码实际上没有维持该不变量
- [ ] 存在性能相当的安全替代方案

**Edition 2024 的 unsafe 变更**——标记之前检查 `Cargo.toml` 中的 edition：
- edition 2024 要求 `unsafe fn` 内使用 `unsafe {}`（不是风格选择）
- edition 2024 要求使用 `unsafe extern "C" {}`——直接使用 `extern "C" {}` 会产生编译错误
- edition 2024 要求使用 `#[unsafe(no_mangle)]` 和 `#[unsafe(export_name)]`
- 在 edition 2021 中，这些模式是可选的风格选择——不要强制要求使用

## 版本感知审查

**在标记任何 edition 特定模式之前**，检查 `Cargo.toml` 中的项目 edition：

```toml
[package]
edition = "2024"  # or "2021", "2018"
```

会影响审查发现的 Edition 2024 变更：

| 变更 | Edition 2021 | Edition 2024 |
|--------|--------------|--------------|
| `unsafe fn` 内的 `unsafe` | 可选风格 | 必需（`unsafe_op_in_unsafe_fn` = deny） |
| `extern "C" {}` | 有效 | 必须使用 `unsafe extern "C" {}` |
| `#[no_mangle]` | 有效 | 必须使用 `#[unsafe(no_mangle)]` |
| `#[export_name]` | 有效 | 必须使用 `#[unsafe(export_name)]` |
| `-> impl Trait` 生命周期捕获 | 仅显式捕获 | 捕获作用域内的所有生命周期 |
| 将 `gen` 用作标识符 | 有效 | 保留关键字（使用 `r#gen`） |
| `!` 类型回退 | 回退到 `()` | 回退到 `!` |
| `if let` 临时值 | 在块末尾丢弃 | 更早丢弃（在 `if let` 末尾） |
| 尾表达式临时值 | 在局部变量之后丢弃 | 在局部变量之前丢弃 |
| `Box<[T]>` 迭代 | 需要显式调用 `.iter()` | 具有 `IntoIterator` impl |

**如果没有指定 edition**，Rust 默认使用 edition 2015。大多数现代项目使用 2021 或更高版本。

**交叉参考**：[rust-code-review](https://github.com/existential-birds/beagle/blob/a5919c1415f2a98780d9da56212d9af52324f88c/plugins/beagle-rust/skills/rust-code-review/SKILL.md) 和 [rust-best-practices](https://github.com/existential-birds/beagle/blob/a5919c1415f2a98780d9da56212d9af52324f88c/plugins/beagle-rust/skills/rust-best-practices/SKILL.md) 技能提供特定于 edition 的代码审查指导和惯用模式。

## 宏专项验证

### “宏卫生问题”

**标记之前**，必须：
1. 验证标识符是否确实泄漏——类型、模块和函数在 `macro_rules!` 中**不**具备卫生性
2. 检查导出的宏是否正确使用 `$crate`（而不是 `crate` 或 `self`）
3. 确认是否需要 `::core::` / `::alloc::` 路径（仅用于在 no_std 上下文中使用的宏）
4. 检查宏是否仅供内部使用，还是使用了 `#[macro_export]`

**常见误报：**
- 内部宏中的非卫生类型名称——仅对导出的宏有影响
- 仅为 `pub(crate)` 的宏未使用 `$crate`——`$crate` 用于跨 crate 使用
- 在仅支持 std 的 crate 的宏中使用 `::std::`——仅当 crate 支持 no_std 时才标记

### “过程宏性能”

**标记之前**，必须：
1. 验证该宏确实位于 proc-macro crate 中（检查 `Cargo.toml` 是否包含 `proc-macro = true`）
2. 检查是否最小化了 `syn` feature（启用 `"full"` feature 的完整 `syn` 与选择性 feature）
3. 确认对编译时间的影响确实有意义（过程宏用于许多文件，而不是一次性使用）

### “错误的片段类型”

**标记之前**，必须：
1. 验证建议的片段类型确实适用于该位置
2. 检查是否为灵活性而有意使用 `:tt`（在 TT munching 模式中很常见）
3. 确认 `:expr` 的贪婪问题确实会出现（使用宏的实际调用点进行测试）

## FFI 专项验证

### “缺少 repr(C)”

**标记之前**，必须：
1. 确认该类型确实跨越 FFI 边界（传入 C 代码或由 C 代码返回）
2. 检查该类型是否仅在 FFI 包装器的 Rust 一侧使用
3. 验证是否改用了 `#[repr(transparent)]` 包装器

**常见误报：**
- 在 FFI 调用前转换的内部 Rust 类型——只有面向 FFI 的类型需要 `repr(C)`
- 与 `repr(transparent)` newtype 包装器一起使用的类型——由包装器处理布局
- 不透明指针类型（`*mut c_void`）——不需要布局保证

### “FFI 安全性”

**标记之前**，必须：
1. 检查 unsafe FFI 调用是否有记录不变量的 SAFETY 注释
2. 验证所有权转移是否确实不明确（检查是否存在成对的 `Box::into_raw`/`Box::from_raw`）
3. 确认 CString 生命周期问题真实存在（CString 的生命周期必须长于传给 C 的指针）
4. 检查回调展开是否确实可能发生（纯数据函数不会跨 FFI 发生 panic）

**常见误报：**
- 永不 panic 的 `extern "C" fn` 回调——不需要 `catch_unwind`
- 在同一作用域内持有来自 CStr::as_ptr() 的 `*const c_char`——生命周期没有问题
- 包含 `unsafe` 的 Bindgen 生成代码——bindgen 输出在设计上本就大量使用 unsafe

## 并发专项验证

### “内存序过弱”

**标记之前**，必须：
1. 验证原子值是否确实在线程之间共享，并且这些线程需要同步
2. 检查 `Relaxed` 是否足够（没有依赖数据的计数器、标志）
3. 确认选择 `Acquire/Release` 还是 `SeqCst` 是否有影响（大多数代码不需要 SeqCst）

**常见误报：**
- 对简单计数器/指标使用 `Relaxed`——独立值不需要排序
- 对循环轮询的布尔标志使用 `Relaxed`——循环会提供最终可见性
- “为了安全”使用 `SeqCst`——这并没有错，只是可能进行了过度同步

## 提交审查之前

**提交门禁**——**通过：** 每个发现都使用 `[FILE:LINE] ISSUE_TITLE`，并包含能够证明问题的确切行（或最小连续行集），以便读者无需相信记忆即可直接跳转到证据。

最终验证：
1. 重新阅读每个发现并询问：“我是否已验证这确实是一个问题？”
2. 对于每个发现，能否指出证明该问题存在的具体行？
3. Rust 领域专家会认同这是一个问题，还是只会认为它是一种风格偏好？
4. 修复此问题是否会提供实际价值，还是只会制造无意义的忙碌工作？
5. 将每个发现格式化为：`[FILE:LINE] ISSUE_TITLE`
6. 对于每个发现，询问：“此修复是在修改现有代码，还是要求添加以前完全不存在的新代码？”如果是后者，将严重级别降为 Informational。
7. 如果这是重新审查：**仅**验证先前的修复。不要提出新的发现。

如果对任何发现存在疑问，请选择以下方式之一：
- 将其从审查中移除
- 将其标记为问题，而不是缺陷
- 阅读更多代码上下文以进行验证
